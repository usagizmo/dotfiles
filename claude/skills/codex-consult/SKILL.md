# Codex への実装方針の事前確認

プランや実装方針を立てたら、コードを書き始める前に Codex にレビューさせ、セカンドオピニオンを取り込んでから進める。

## なぜこのスキルが必要か

一人で立てたプランには、立てた本人には見えない盲点が必ず残る。前提の取り違え、考慮漏れのエッジケース、より筋の良い代替設計、見落としたトレードオフ — これらは「別の頭」に通すと一気に表面化する。

Codex はちょうどその「別の頭」になる。Claude とは別系統の独立したモデルなので、異なる発想の癖を持ち込める。Codex の `AGENTS.md` は Claude の `CLAUDE.md` へのシンボリックリンクだが、`CLAUDE.md` 本体は短く、詳細な設計原則や `refine`/`tidy` の中身までは渡らない。だから**評価の判断軸は、このスキルのプロンプトに明示的に書き込む**（下記参照）。北極星は `refine`（ゼロベース設計）と `tidy`（根本解決・クリーン化）でかけている基準そのもの。

目的は「Codex に従う」ことではなく、**実装に着手する前に方針を 1 段強くする**こと。Codex の指摘は精査の対象であって、命令ではない。

## いつ発動するか

プランニング段階すべて。具体的には:

- 新機能・新モジュールの設計方針を立てたとき
- リファクタリング・改善のアプローチを決めたとき
- バグ修正の原因仮説と修正方針を立てたとき
- 複数の実装方法でトレードオフを比較するとき
- Plan Mode でプランをまとめ、ユーザーに提示する前

「どう実装するか」を考えたら、コードを書く前に一度ここを通す。Plan Mode で動いている場合は、`ExitPlanMode` でプランを提示する直前にこの確認を挟む。

## 実行手順

### 1. まず Claude 自身のドラフト方針を立てる

Codex に丸投げしない。まず自分で実装方針のドラフトを作る。これがレビュー対象になる。ドラフトには最低限:

- 何を作る/変えるのか（要件・ゴール）
- どのファイル・モジュールをどう変えるのか
- 主要な設計判断とその理由
- 想定するエッジケース・懸念点（自分で気づいている範囲で）

を含める。ドラフトが曖昧なほど Codex のレビューも曖昧になるので、自分の中で筋を通してから渡す。

### 2. プロンプトをファイルに書き出す

プロンプトにはコードや記号が混じり shell のクォートで壊れやすいため、**一時ファイルに書いてから stdin で渡す**。`Write` ツールで `/tmp/codex-consult-prompt.md` に以下の構成で書く:

```markdown
あなたは実装方針のレビュアーです。以下のプランをレビューし、セカンドオピニオンを返してください。
コードの変更はしないでください。分析と提案だけを返してください。

## 背景・要件
{何を作る/変えるのか。ゴールと制約}

## 関連ファイル
{Codex が読むべきファイルパス。Codex はリポジトリを read-only で読めるので、
 全文を貼らずパスと「どこに注目すべきか」を示す}

## 私のドラフト方針
{Claude が立てた実装方針。設計判断とその理由を含める}

## レビューしてほしい観点
最重要の判断軸（トップエンジニアの目線）:
- **ゼロベース設計か**: この方針は、実装の経緯を忘れてゼロから設計したときに辿り着く最適解と一致するか。既存実装に引きずられたパッチ的な方針になっていないか
- **根本解決か**: 症状を抑えるだけの継ぎ足しでなく、原因に対する抜本的な設計か。根本解決のためなら改修スコープを広げる前提でよい
- **クリーンか**: 互換のための温存コード・deprecated alias・古い実装・dead code を残す前提になっていないか。新設計でクリーンに置き換える方針か

加えて:
- 見落とし・考慮漏れのエッジケースはないか
- より筋の良い代替設計はないか（あれば理由とともに）
- 隠れたトレードオフ・リスクはないか

末尾に、方針への総合評価（このまま進めてよい / 修正を推奨 / 再考を推奨）を一言で。
重箱の隅つつき (nitpick) は不要。実装の成否を左右する指摘に絞ってください。
```

### 3. Codex に渡す

read-only サンドボックスで実行する（Codex はコードを読むだけで書き換えない）。最終回答だけを `-o` でファイルに書き出すと、思考ログが混ざらずクリーンに受け取れる:

```bash
codex exec -s read-only -o /tmp/codex-consult-out.md - < /tmp/codex-consult-prompt.md
```

- `-s read-only`: Codex はリポジトリを読むだけ。誤って変更させない安全策
- `-o <file>`: Codex の最終回答だけをファイルへ。stdout の冗長ログを避ける
- `- < file`: プロンプトを stdin から読む（クォート事故を防ぐ）
- モデルと reasoning effort はコマンドで指定せず、codex の config に従う。セカンドオピニオンは質が命なので effort を下げる方向の上書きはしない
- reasoning が深いと応答に 1〜数分かかることがある。実行前にユーザーへ「Codex に方針を確認します」と一言伝えてから走らせる

実行後、`Read` ツールで `/tmp/codex-consult-out.md` を読む。

### 4. 一致点・相違点を整理して統合する

Codex の回答を鵜呑みにも黙殺もしない。**Claude のドラフトと突き合わせ**、各論点を 3 つに仕分ける:

- **一致**: Codex も同意 → 方針の確度が上がった。そのまま採用
- **相違・追加指摘**: Codex が別案・懸念を出した → 中身を精査する。妥当なら方針に取り込む。妥当でない（誤検知・前提のズレ）なら、なぜ採用しないかを言語化する
- **判断が割れる**: どちらの設計にも理がある／プロダクトの方向性に依存する → ユーザーに判断を仰ぐ

### 5. 統合した方針を提示する

ユーザーには以下を簡潔に示す:

```markdown
## 実装方針（Codex 確認済み）

{統合した最終方針}

### Codex のレビューを反映した点
- {取り込んだ指摘とその理由}

### Codex の指摘で採用しなかった点（あれば）
- {指摘 → 採用しない理由}

### 判断を仰ぎたい点（あれば）
- {割れた論点と選択肢}
```

判断が割れる重要な相違があれば、そこはユーザーの決定を待ってから着手する。なければ統合方針で実装に進む。

## Codex が使えないとき

- `codex: command not found` → Codex 未インストール。ユーザーに伝え、セカンドオピニオンなしで進めてよいか確認する
- 認証エラー → `codex login` が必要。ユーザーに案内する
- それ以外のエラー → エラー内容を伝え、Claude 自身のドラフト方針で進める（Codex が落ちてもプランニングは止めない）

いずれの場合も、確認を取れなかった事実は隠さずユーザーに伝える。
