## 変更内容の把握方法
`git diff --cached` と `git diff` を実行。
diff が空なら `git log -1 --format="" --name-only` で直近コミットの変更ファイルを確認し、`git diff HEAD~1` で差分を取得。

## 確認対象

### 1. プロジェクト CLAUDE.md
プロジェクト全体に関わる重要な変更:
- コーディング原則・規約の変更
- 新しいパターンやルールの追加
- 開発コマンドの変更
- スキル一覧の更新

### 2. プロジェクト Rules（`.claude/rules/*.md`）
- 新しい規約の追加が必要か
- 既存の規約に変更が必要か
- 古くなった規約の削除

### 3. プロジェクト Skills（`.claude/skills/`）
- description の適用タイミングが正確か
- 手順や形式に変更が必要か
- 新しいスキルの追加が必要か
- リファレンスの更新が必要か

### 4. グローバル Rules（`~/.claude/rules/*.md`）
- プロジェクト横断で適用すべき新しい規約か

### 5. グローバル Skills（`~/.claude/skills/`）
- プロジェクト横断で使うスキルに変更が必要か

### 6. セッション知見の記録先判断

| 条件 | 記録先 |
| --- | --- |
| プロジェクト固有の規約・パターン | プロジェクト rules / skills |
| プロジェクト横断の規約・パターン | グローバル rules / skills |
| どちらにも該当しない | 記録不要 |

### 7. 設計振り返り（Design Retrospective）

セッション全体の変更を俯瞰し、設計レベルの学びを抽出してください。
（/refine のような個別コードレビューではなく、セッションから蒸留すべき汎用的な設計原則の発見）

#### 問い
以下の観点について、今回のセッションで「美しい設計」に近づいた判断、または離れた判断があったか振り返ってください:

- **責務分割**: モジュール・関数の責務は単一か。「〜と〜をする」になっていないか
- **データフロー**: データの変換経路は最短か。状態の持ち主は適切か（SSOT）
- **抽象化レベル**: 粒度は適切か。同じレイヤーに同じ抽象レベルの処理があるか
- **パターン整合性**: 既存パターンと整合しているか
- **API 設計**: 呼び出し側にとって自然なインターフェースか

#### 蒸留すべきもの
- 「なぜこの設計が美しいか」の原則化（回避的ではなく追求的に記述）
- プロジェクト固有の設計美学
- アンチパターンとその理由（なぜダメか、何が代わりに正解か）

#### 記録先の判断

| 内容 | 記録先 | 制約 |
| --- | --- | --- |
| プロジェクト横断で適用できる蒸留された設計原則 | `~/.claude/rules/design-principles.md` | 具体例ではなく原則のみ。既存と重複する場合は統合 |
| プロジェクト固有の設計知見（パターン・アンチパターン・ドメイン知識） | 該当する `.claude/skills/*/references/` | 既存ファイルがあれば追記、なければ新規作成 |

#### 記録しない
- セッション固有の一時的な判断（パターンとして確立していないもの）
- コードから自明に読み取れる設計意図
- 既に記録済みの原則の繰り返し

#### 書き方
- 「〜してはいけない」ではなく「〜する方が美しい。なぜなら〜」の形式を優先
- 具体例は ✅ Good / ❌ Bad の対比（既存スタイルに合わせる）

## 更新判断基準

**記載する:**
- プロジェクト固有の設計判断とその理由
- 判定フロー（いつ・何を使うか）
- 試行錯誤の末に確立したパターン（罠・ハマりポイント含む）
- エージェントが次回同じ間違いをしないための指針

**記載しない:**
- Claude が一般知識として持つ標準パターン
- コードや型定義から推測可能な情報
- 一般的な技術説明
- 一時的な実装でパターンとして確立していないもの

## 更新原則
1. **簡潔に**: 実装と二重管理にならないよう、要点のみ
2. **最新化**: 古くなった情報・項目はすべて削除
3. **大胆に**: より良い構成が思いついたら積極的に変更

## 出力形式
- 更新が必要なファイル: パス・変更内容を具体的に
- 更新不要の場合: 「更新不要」とだけ回答
