# Codex レビューループ

実装を書き終えたら、Codex に差分をレビューさせ、指摘を精査して修正し、再レビューする。これを完走するまで繰り返し、Codex が納得する（または残る指摘を正当な理由で棄却できる）状態まで持っていく。

## なぜこのスキルが必要か

自分が書いたコードのバグは、自分では見つけにくい。「こう書いたつもり」というバイアスが、実際の挙動とのズレを覆い隠す。Codex に差分を渡すと、その思い込みの外側からバグ・抜け・設計のほころびを指摘してくれる。

Codex の `AGENTS.md` は Claude の `CLAUDE.md` へのシンボリックリンク。ただし `CLAUDE.md` 本体は短く、詳細な設計原則（`design-principles.md`）や `tidy` の中身までは Codex に渡らない。だから**レビューの判断軸は、このスキルのプロンプトに明示的に書き込む**。

レビューの北極星は 3 つ:

1. **ゼロベース設計との一致** — 実装の経緯を忘れ、「今日この要件を受け取ったトップエンジニアがゼロから設計したら同じ形になるか」
2. **パッチでなく根本解決** — 小手先の継ぎ足しで症状を抑えただけになっていないか。原因への抜本的な改善になっているか。根本解決のためなら改修スコープを広げることを厭わない
3. **互換コードの完全排除** — 後方互換のためだけの記述・deprecated alias・互換 shim・古い実装の温存・dead code が残っていないか。新しい設計でクリーンに置き換わっているか（`tidy` の核心）

普段「ゼロベース設計」と `tidy`（クリーン化）でかけている基準を、そのまま Codex レビューにも持ち込む。

ただし Codex の指摘は「命令」ではなく「精査の対象」。妥当な指摘は直し、誤検知や前提のズレによる指摘は理由を添えて棄却する。**指摘に機械的に従ってコードを歪めない** — これも設計原則の「中途半端な併存を選ばない／根本的な改善を優先」と同じ姿勢。

## いつ発動するか

まとまった実装を終えたタイミングすべて:

- 機能実装・バグ修正・リファクタを一通り書き終えたとき
- 「実装できた」「動くようになった」と判断したとき
- コミットする前（`/tidy` → `/commit` の前段）

## 実行手順

### 1. レビュー対象（diff のスコープ）を決める

`git status --short` と `git branch --show-current` で状況を確認し、Codex に何を diff させるか決める:

- **未コミットの変更がある** → working tree をレビュー（`git diff` + `git diff --cached` + untracked）。実装直後の典型ケース
- **変更がコミット済みで、フィーチャーブランチにいる** → ベースブランチとの差分をレビュー（`git diff <base>...HEAD`）

判断に迷う場合はユーザーに確認する。決めたスコープは、次のプロンプトに「どう diff を取るか」として書き込む。

### 2. レビュー依頼のプロンプトをファイルに書き出す

プロンプトには記号が混じり shell のクォートで壊れやすいため、**一時ファイルに書いてから stdin で渡す**。`Write` ツールで `/tmp/codex-review-prompt.md` に以下の構成で書く（スコープに応じて diff コマンドを差し替える）:

```markdown
あなたはコードレビュアーです。以下の変更をレビューしてください。コードは変更しないでください。

## レビュー対象
まず次のコマンドで変更内容を把握してください:
{未コミットなら: `git status --short` と `git diff` と `git diff --cached`、untracked ファイルは中身も確認}
{ベースブランチ比較なら: `git diff <base>...HEAD`}

## レビュー観点
最重要の判断軸（トップエンジニアの目線で厳しく見てください）:
1. **ゼロベース設計との一致**: 実装の経緯を一切忘れ、「今日この要件を受け取ったトップエンジニアがゼロから設計したら同じ形になるか」。ならない箇所は、なぜズレているかと理想形を示してください
2. **パッチでなく根本解決**: 小手先の継ぎ足し・条件分岐の追加で症状を抑えただけになっていないか。原因に対する抜本的な改善になっているか。根本解決のためなら改修スコープを広げる提案を厭わないでください
3. **互換コードの完全排除**: 後方互換のためだけの記述・deprecated alias・互換 shim・古い実装の温存・dead code が残っていないか。新しい設計でクリーンに置き換わっているか

加えて、具体的な不具合も探してください:
- バグ・ロジックの誤り・考慮漏れのエッジケース
- 軸の混在 / SSOT 違反 / 責務の置き場所が誤っている箇所
- 規約違反（型の緩さ、命名 等）

## 出力形式
発見した問題を重要度順に箇条書きで、各項目を「ファイル:行 / 問題 / 推奨修正」の形式で簡潔に。
問題がなければ「指摘なし」とだけ返してください。
重箱の隅つつき (nitpick) は避け、実害のある指摘に絞ってください。
```

### 3. Codex にレビューさせる

read-only サンドボックスで実行し、最終回答だけを `-o` でファイルに書き出す（思考ログが混ざらずクリーンに受け取れる）:

```bash
codex exec -s read-only -o /tmp/codex-review-out.md - < /tmp/codex-review-prompt.md
```

- `-s read-only`: Codex は diff を読むだけ。誤って変更させない安全策
- `-o <file>`: Codex の最終所見だけをファイルへ。stdout の冗長ログを避ける
- `- < file`: プロンプトを stdin から読む（クォート事故を防ぐ）
- モデルと reasoning effort はコマンドで指定せず、codex の config に従う。レビューは質が命なので effort を下げる方向の上書きはしない
- reasoning が深いと応答に 1〜数分かかることがある。実行前にユーザーへ「Codex にレビューを依頼します」と一言伝える

実行後、`Read` ツールで `/tmp/codex-review-out.md` を読む。「指摘なし」なら完走（ステップ 6 へ）。

### 4. 各指摘を深く精査する

ここがこのスキルの核心。指摘を流し読みで直さない。1 件ずつ:

- **本当にバグ/問題か** — 該当コードを `Read` で実際に確認する。Codex が差分だけ見て前提を取り違えていないか、呼び出し元まで辿って検証する
- **直す価値があるか** — 実害のある指摘か、単なる好みか。上記 3 本柱（ゼロベース設計 / 根本解決 / 互換コード排除）に照らして判断する
- **どう直すのが正しいか** — その場しのぎのパッチではなく、根本的に正しい修正を考える。互換のための温存コードは残さず削除し、新しい設計でクリーンに置き換える。根本解決のためなら改修スコープを広げてよい。ただし修正が当初の変更と無関係な大規模リファクタに膨らむ場合は、boy-scout の「スコープ判断」に従い、いったんユーザーに相談する（同じ PR で解決すべきか、別 Issue に切り出すか）

指摘は 3 つに仕分ける:

- **採用**: 妥当な指摘 → 根本的に正しい形で修正する
- **棄却**: 誤検知・前提のズレ・好みの問題 → なぜ直さないかを言語化して記録（後でユーザーに報告するため）
- **判断が割れる**: プロダクトの方向性に依存する／どちらの設計にも理がある → ループを止めてユーザーに相談する

### 5. 修正して再レビュー

採用した指摘を修正したら、変更後の diff に対してステップ 2〜4 を繰り返す。修正でコードが壊れていないか（型・lint・テストが軽量に回るなら）も確認してから次のレビューへ。

### 6. 完走判定

以下のいずれかで完走とする:

- **Codex が「指摘なし」を返した** — クリーン
- **残る指摘がすべて、精査の結果「誤検知/不要」と結論づけられる** — これ以上直す必要がない

無限ループを避けるため、**最大 5 ラウンド**を安全上限とする。Codex が毎回新しい nitpick を出し続けて収束しない場合、5 ラウンドで打ち切り、残った指摘と「なぜ収束しなかったか」をユーザーに報告する。修正が別の指摘を生んで堂々巡りになっていないかも監視する。

### 7. 完走後の報告

ユーザーに簡潔に報告する:

```markdown
## Codex レビューループ完了（{N} ラウンド）

### 修正した指摘
- {指摘 → どう直したか}

### 棄却した指摘（あれば）
- {指摘 → なぜ直さなかったか}

### 未解決（上限到達時のみ）
- {残った指摘と状況}
```

判断が割れた指摘があれば、ループの途中でユーザーに相談済みのはず。完走後はそのままコミット（`/tidy` → `/commit`）に進める状態にしておく。

## Codex が使えないとき

- `codex: command not found` → 未インストール。ユーザーに伝え、レビューなしで進めてよいか確認する
- 認証エラー → `codex login` が必要。ユーザーに案内する
- それ以外のエラー → エラー内容を伝え、レビューを取れなかった事実を隠さず報告する
