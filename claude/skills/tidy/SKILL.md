## 変更内容の把握方法
`git diff --cached` と `git diff` を実行してください。
diff が空の場合は `git log -1 --format="" --name-only` で直近コミットの変更ファイルを確認し、`git diff HEAD~1` で差分を取得してください。

## レビュー観点

### 設計の妥当性
> トップエンジニアが今日、一から設計したとしても同じ設計にするか？
「いいえ」なら、クリーンな設計を提案する。

### ボーイスカウトルール
変更ファイルとその周辺で `~/.claude/rules/boy-scout.md` に記載の改善を確認・適用する。
今回の変更に関係ない箇所でも、目に入った改善は積極的に行う。

### 周辺との整合性
変更内容が周辺のコード・設定と矛盾していないか確認する。
同じパターンやルールを使っている兄弟ファイルがあれば、そちらにも同じ変更が必要でないか確認する。

### 追加の確認観点
- `TODO: remove after migration` 等のコメントが付いた暫定コード
- 後方互換のためだけの型エイリアス・ラッパー・re-export・`@deprecated`
- 使われていないフィーチャーフラグ
- 過去の制約に基づいた回避策や workaround
