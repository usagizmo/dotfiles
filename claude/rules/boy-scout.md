# ボーイスカウトルール

編集したコードの周辺を、着手前よりも綺麗な状態にする。

## 許容する改善

以下の改善は積極的に行う:

- 型の厳密化（`any` → 適切な型）
- 不要な変数・インポートの削除
- 命名の改善（より意図が伝わる名前へ）
- 早期リターンへの変換
- マジックナンバーの定数化
- 軽微なコードスタイルの統一
- 互換性維持のためだけの古い記述の削除（未使用の再export、deprecatedコード等）
- 不要なコード・ファイルの削除（dead code、空ファイル、スタブのみのファイル等）
- 周辺コードの共通化・抽象化（重複ロジックの統合、共通パターンの抽出）

## スコープ判断

周辺改善の変更が大きくなりそうな場合:

1. まず一定の調査を行い、改善の全体像と影響範囲を把握する
2. 現在のスコープ内で解決できるなら実施する（調査結果は Issue 化する場合にも活用できる）
3. 明らかにスコープを超える場合は、調査結果をもとに GitHub Issue の作成を提案する

## 報告

改善した箇所は完了時に簡潔に報告する。
