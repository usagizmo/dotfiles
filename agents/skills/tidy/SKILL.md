# コードクリーンアップ

差分の設計妥当性は `consult` / `review-loop`。ここはクリーンアップと周辺改善。

## 契約

| 誰 | すること |
| --- | --- |
| レビュアー（subagent 可） | 変更せず指摘だけ返す。harness 固有 API は指定しない |
| 親 | 指摘を適用ゲートで再分類し、適用・検証・報告する。subagent が使えなければ同じ観点で自己レビューする |

## 範囲

| 軸 | 意味 |
| --- | --- |
| 起点 | 今回触ったファイル。混在 working tree は絞る。起点一覧と diff の食い違いは報告する |
| 調査 | 周辺・caller・兄弟を制限なく読む |
| 修正 | 起点外でも直す。完全に別テーマで独立レビューが要る規模だけ Issue / 別作業へ |

## 適用ゲート（親）

レビュアーのラベルをそのまま見送りにしない。

**即適用**（「提案」とあっても。分量増・プラン未記載は理由にしない）:

- 起点 chokepoint / 契約 / SQL / API の SSOT 複製（test が production 形を手コピー、同一 helper の二重）
- 今回差分と周辺の shape / 型 / mock / 生成物 drift
- CI / lint / formatter 失敗
- 同一テーマの局所 boy-scout（dead 削除、命名、重複統合）

**見送り可**:

- dead / 実害が caller chain でも判別不能
- 完全に別テーマで独立レビューが要る規模

**規模再判定（見送りではない）**:

- 直すと非自明な挙動変更やスコープ拡大になる → 適用前に規模を再判定し、中規模 / 大規模の仕上げフローへ戻す（即適用と衝突したら再判定が先）

**見送り禁止**:

- 仕上げを急ぐ
- プラン / issue に無い、またはプランの検証方針と違う
- 「スコープ拡大」だけ（起点同一 chokepoint の SSOT 化・契約揃えは拡大ではない）
- レビュアーが「提案」と書いたから

## 手順

1. `git status --short` で起点を確定する
2. 下記プロンプトでレビューする（自己レビュー可）
3. 適用ゲートで再分類する。即適用を直す。規模再判定が要れば先に戻す。見送りは理由を 1 行残す
4. 最小の関連検証を可能な範囲で行う
5. 修正と見送りを簡潔に報告する

## 観点
- 互換 shim / dead code / 1-variant・将来用 abstraction を温存しない
- dead 判定は caller chain を実コードで辿る。判別不能なら提案に留める
- 起点 chokepoint の production 形を test / 別層が手コピーしている SSOT 複製 → 修正すべき
- 今回差分と周辺の shape / 型 / mock / 生成物 drift、兄弟の同修正要否

- 修正すべき: パス・行・内容
- 提案（要確認）: 判別不能・要確認に限る（SSOT 複製・drift・CI 失敗をここに入れない）
