# コードクリーンアップチェック

実装者のバイアスを排除するため、サブエージェントが使える環境では第三者視点でレビューさせ、使えない環境では同じ観点で自己レビューする。

差分そのものの設計の妥当性（責務分割・データフロー・API 形状）に不安がある場合は、この skill で吸収せず `consult`（設計の問い直し）または `review-loop`（差分レビュー）にエスカレートする。

## サブエージェント契約

- **役割**: レビュー中は変更しない。指摘と提案だけを返す
- **適用・検証・報告**: 親エージェントが行う
- **モデル指定なし**: 汎用サブエージェント。harness 固有の agent 種別・起動 API は共通 skill では指定しない
- サブエージェントが使えない環境では、同じチェックリストで親が自己レビューする

## 起点・調査・修正の範囲

| 軸 | 意味 |
| --- | --- |
| **起点** | 今回のセッションで触ったファイル（authoritative。ここからレビューを始める） |
| **調査** | 判断のため read-only で読む周辺・caller・兄弟ファイル。制限しない |
| **修正** | 起点に加え、調査で見つけた改善も積極的に直す（boy-scout）。目的タスクだけに閉じない |

- より良い書き方・dead code・整合の崩れを見つけたら、起点外でも直す。無視してもったいない
- テーマが完全に別・影響が広く独立レビューすべき規模だけ、適用せず「別作業向け提案」に回す（`boy-scout.md` のスコープ判断）
- 起点一覧と diff が食い違うときは勝手に起点を増やさず、食い違いを報告する（不明ならユーザーに確認）
- working tree に今回以外の変更が混在している場合は起点を絞る。判断できなければユーザーに確認する
- diff が空のときの直近コミット fallback は、親が「直近コミットが対象」と明示した場合だけ使う（プロンプトにもその旨を書く）

## 調査と修正の方針
- 判断に必要な周辺・caller chain・兄弟ファイルは自由に読んでよい
- 起点外でも、より良い書き方・dead code・不整合を見つけたら指摘する（無視しない）
- テーマが完全に別で大規模なものは「別作業向け提案」に分け、それ以外は「修正すべき」に含める

## 変更内容の把握方法
1. `git status --short` を実行する
2. 起点内の tracked 変更は `git diff --cached` と `git diff` で把握する（パスを絞れるなら絞る）
3. 起点内の untracked ファイルは中身を直接読む
4. diff が空で、かつ親が「直近コミットが対象」と明示している場合のみ `git diff HEAD~1` を使う

## レビュー観点

（差分そのものの設計の正しさは `review-loop` の担当。この skill はクリーンアップと周辺改善に特化する）

### 判断軸
- 互換 shim、dead code、使われない抽象を温存しない
- dead 判定は参照検索だけでなく caller chain を実コードで辿る。判別不能なら削除を断定せず提案に留める
- 1-variant に落ちた dispatch や将来用 abstraction を削除する
- 関連改善は同じ変更でまとめて直す。独立した大規模変更だけ別提案にする
- 具体シグナルの例: `TODO: remove after migration`、互換だけの型エイリアス・ラッパー・re-export・`@deprecated`、使われていないフィーチャーフラグ、過去制約の workaround

### 周辺との整合性
起点の変更が周辺と矛盾していないか、兄弟ファイルに同じ修正が必要でないかを確認し、必要ならそれも指摘する。

## 出力形式
- 修正すべき箇所: ファイルパス・行番号・修正内容（起点外の改善も含む）
- 不確実なものは「提案（要確認）」と明記
- テーマが別の大規模改善だけ「別作業向け提案」として分ける
- 問題なしの場合: 「問題なし」とだけ回答
