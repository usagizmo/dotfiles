# ドキュメント更新チェック

次回以降のセッションでエージェントがより正確に動けるよう、ドキュメントを最新化する。
役割は「新しい設計原則の発見」ではなく、**確立済みのプロジェクト固有知見の永続化判定**と、触ったドキュメント周辺の削除・最新化である。
実装者の「自分はわかっているから不要」バイアスを排除するため、サブエージェントが使える環境では第三者視点で評価させ、使えない環境では同じ観点で自己レビューする。

追加を検討する前に、必ず「削れる箇所」を先に洗い出す。観点は `references/review-prompt.md` に集約している。

## サブエージェント契約

- **役割**: レビュー中は変更しない。評価と提案だけを返す
- **適用・報告**: 親エージェントが行う
- **モデル指定なし**: 汎用サブエージェント。harness 固有の agent 種別・起動 API は共通 skill では指定しない
- サブエージェントが使えない環境では、同じテンプレートで親が自己レビューする

## 対象ドキュメント

プロジェクト管理下のエージェント向けドキュメント（所有スコープ）。ホーム側の `~/.agents/` は対象外。

| 種類 | 何を書くか |
| --- | --- |
| **AGENTS.md**（リポジトリ根やプロジェクト規約上の入口） | プロジェクト概要・コマンド・スキル一覧 |
| **Rules**（プロジェクトの rules ディレクトリ） | 全セッションで適用される規約・原則 |
| **Skills**（プロジェクトの skills ディレクトリ） | 特定タスク用の手順・判断基準・リファレンス |
| **ユーザー向けドキュメント** | README / docs サイト等。機能変更と矛盾していれば直す |

配置パスはプロジェクトごとに異なる（`.agents/` とは限らない）。既存のプロジェクト規約があればそれに従う。

## 起点・調査・更新の範囲

| 軸 | 意味 |
| --- | --- |
| **起点** | 今回のセッションで触ったファイル（authoritative） |
| **調査** | 既存の AGENTS / rules / skills / ユーザー向け docs を read-only で照合。未変更ファイルも読んでよい |
| **更新** | 起点に対応する記述に加え、調査で見つけた重複・陳腐化・不整合も直す。目的タスクの最小差分に閉じない |

- より良い書き方・削れる記述を見つけたら、起点外のドキュメントでも直す
- テーマが完全に別・大規模な再構成だけ「別作業向け提案」に回す
- 起点一覧と diff が食い違うときは勝手に起点を増やさず、食い違いを報告する（不明ならユーザーに確認）
- working tree に今回以外の変更が混在している場合は起点を絞る。判断できなければユーザーに確認する
- diff が空のときの直近コミット fallback は、親が「直近コミットが対象」と明示した場合だけ使う（プロンプトにもその旨を書く）

## 実行手順

1. **起点を確定する**（`git status --short`。untracked も含める）
2. セッション中の材料を整理する（切り分けは排他）:
   - **知見（ヒント）**: 検証可能な罠・ハマり・正解パターン。設計判断の理由はここには書かない
   - **設計上の選択（主張）**: 「A ではなく B を選んだ。理由は〜」。照合対象であり、これ単独で一般原則にしない
3. サブエージェントが使える環境では、`references/review-prompt.md` に以下を埋めて渡す。使えない環境では同じテンプレートで自己レビューする（評価まで。適用は手順 4）:
   - 起点のファイルパス一覧
   - 知見（ヒント）
   - 設計上の選択（主張）
   - 直近コミット fallback を使う場合はその旨
4. 結果を精査し、妥当な更新は起点外でも適用する
5. 更新・見送った箇所を簡潔に報告する
