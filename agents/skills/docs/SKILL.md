# ドキュメント更新チェック

次回以降のセッションでエージェントがより正確に動けるよう、ドキュメントを最新化する。
加えて、セッション全体を俯瞰した設計振り返りを行い、「次回から指示なしに美しい設計を選べる」ための原則を蒸留・永続化する。
実装者の「自分はわかっているから不要」というバイアスを排除するため、サブエージェントが使える環境では第三者視点で評価させる。使えない環境では、同じ観点で自己レビューする。

## 削除を先に検討する（追加バイアスへの対抗）

ドキュメントは追記されやすく削除されにくく、放置すると密度が下がり読まれなくなる。
追加を検討する前に、必ず先に「削れる箇所」を洗い出す。差分が `+` ばかりになっていたら設計が間違っている可能性を疑う。

- 同じ概念が複数ドキュメントに現れていないか → 実装または 1 つの skill を SSOT にし、他は最小限の参照に置き換える
- 経緯説明・移行 note・"以前は〜" の言及 → 現状の規約だけ残して経緯は削る
- 列挙が膨らんで冗長になったチェックリスト → 上位原則に集約できないか
- 「念のため」書かれた duplicate な caveat / 冗長な前置き

## 対象ドキュメント

| 種類 | スコープ | 何を書くか |
| --- | --- | --- |
| **AGENTS.md** | 常時読み込み | プロジェクト概要・コマンド・スキル一覧 |
| **Rules** (`.agents/rules/`) | 常時読み込み | 全セッションで適用される規約・原則 |
| **Skills** (`.agents/skills/`) | 関連時に読み込み | 特定タスク用の手順・判断基準・リファレンス |

※ グローバル（`~/.agents/`）とプロジェクト（`.agents/`）の両方が対象。エージェント固有の互換入口（例: `~/.claude/CLAUDE.md`）は共通ファイルへの symlink として扱う。

## 実行手順

1. 今回のセッションで変更したファイルパスを把握する
2. セッション中の知見を整理する:
   - ハマりポイント・試行錯誤の末に見つけた正解パターン・設計判断の理由
   - セッション全体を通じて行った設計上の選択とその根拠
3. サブエージェントが使える環境では、この skill の `references/review-prompt.md` のテンプレートに以下を埋めて渡す。使えない環境では、同じテンプレートで自己レビューする:
   - `{変更したファイルパスの一覧}`
   - `{ハマりポイント・正解パターン・設計判断の理由}`
   - `{設計上の選択とその根拠}`
4. サブエージェントの結果に基づき、ドキュメントを更新する
5. 更新した箇所を簡潔に報告する
