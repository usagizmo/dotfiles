# ドキュメント更新チェック

確立済みのプロジェクト固有知見の永続化判定と、触った docs 周辺の削除・最新化。新しい設計原則の発見器ではない。
追加の前に削れる箇所を先に洗う。観点は `references/review-prompt.md`。

## サブエージェント契約

- レビュー中は変更しない。適用・報告は親
- harness 固有の agent API は指定しない
- 使えない環境では同じテンプレートで自己レビュー

## 対象

プロジェクト管理下のエージェント向け docs。`~/.agents/` は対象外。

| 種類 | 何を書くか |
| --- | --- |
| AGENTS.md | 概要・コマンド・スキル一覧 |
| Rules | 常時適用の規約 |
| Skills | 特定タスクの手順・判断・リファレンス |
| ユーザー向け docs | README / docs サイト等。機能と矛盾していれば直す |

配置はプロジェクト規約に従う。

## 起点・調査・更新

| 軸 | 意味 |
| --- | --- |
| 起点 | 今回触ったファイル |
| 調査 | 既存 AGENTS / rules / skills / ユーザー向け docs を照合。未変更も読んでよい |
| 更新 | 対応記述に加え、重複・陳腐化・不整合も直す |

起点外でも削れる・直せる記述は直す。テーマが完全に別の大規模再構成だけ別提案。起点と diff の食い違いは報告。混在 working tree は起点を絞る。

## 手順

1. `git status --short` で起点を確定
2. 材料を整理する: **知見**は検証可能な罠・正解パターン。**設計上の選択**は「A ではなく B」の主張で照合対象。単独で一般原則にしない
3. `references/review-prompt.md` に起点・知見・選択を埋めて渡す。使えない環境では自己レビュー
4. 妥当な更新は起点外でも適用
5. 更新・見送りを簡潔に報告

## 品質パスの適用・検証

品質基準の SSOT は `~/.agents/rules/design-principles.md` の「品質パス」。ここは手順だけ。

1. 原則を先に読む。文書の都合で原則を緩めない
2. 圧縮対象は節単位で書き直す。太字剥がし・言い換え削除だけの diff は不合格
3. 大型 reference（おおよそ 300 行超）は第 1 パスのあと、見出し一覧と壊れた表 / fence / 目次を再読する第 2 パスを必須にする
4. 完了判定: 判断に必要な情報の被覆が維持または改善され、より短く明確。行数減だけ・構造破損は未完了。非自明な境界の API から動く正例が消えていないかも見る
5. プロジェクトに Markdown lint（目次・識別子 backtick 等）があれば通す
