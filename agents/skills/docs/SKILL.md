# ドキュメント

## いつ

| トリガー | やること |
| --- | --- |
| AGENTS / rules / skills / prompts / references を新規・編集した | コミット前に品質パスを実行する。規模不問。下書きのままで commit しない |
| 中規模以上の仕上げ | project docs 更新判定。agent-facing を触っていれば品質パスも通す |

規模 SSOT は `~/.agents/AGENTS.md`。設計の問い直しは `consult`。

## project 差分（`docs-project`）

available skills に `docs-project` があれば先に invoke し、本 skill への project 差分として適用する。差分は追加・具体化のみで、本 skill の基準・完了条件を緩めない。project 側は `docs-project` 固定名で置き、単体では invoke しない。

## 契約

- 下書きは判断材料の抜け漏れを優先してよい。品質は下記の品質パスが担保する
- 完了はトリガー別（下表）。片軸だけやって docs 完了としない。未完了なら commit しない
- レビュー中は変更しない。適用・報告は親
- harness 固有の agent API は指定しない
- subagent が使えなければ同じ観点で自己レビューする

### 完了報告（トリガー別）

| 該当 | 必須の自己判定 |
| --- | --- |
| agent-facing を触った | 品質パス: pass |
| 中規模以上の仕上げ | project docs 更新判定: 更新済み / 更新不要 |
| 両方 | 上の両方 |

## 品質パス（基準 + 手順）

モデルに読ませる文（agent-facing）の品質 SSOT。rules / AGENTS / skills（`references/` 含む）/ prompts など、形式を問わず適用する。global（`~/.agents/` / dotfiles agents）も project も同じ。

下書きを本パスで基準へ適合させる。

### 基準（成果物）

- **抜け漏れなく、要点だけ**: 判断に必要な不変条件・境界・例外は落とさない。言い換えの重複・経緯・メタ説明は書かない
- **常駐層は薄く**: 常時ロードされる AGENTS / rules に置くのは、ワークフローゲート・恣意的な規約（命名・境界・優先順位の決定）・SSOT への誘導だけ。モデルが自力で再導出できる一般ベストプラクティスの中身は書かない。方向性を定める文（基準の高さ・どちらに倒すか）は恣意的な決定なので残し、挙動を変えない飾り文だけ削る。事故由来の知見・詳細索引は skill / references（遅延ロード）に置き、常駐には発動トリガーを 1〜2 行だけ残す
- **skill 内も同じ極性**: SKILL.md は発動条件・判断・SSOT 誘導に留め、状況限定の手順詳細・事例は `references/` へ。ただし数行で書ける知見は reference を新設せず本文で圧縮する
- **行動を変えられる具体性を持たせる**: 何をするか / 何をしないか / SSOT はどこかを明示する。装飾や自己言及は削る
- **非自明な境界を持つ API は動く正例を残す**: prose の言い換えで代替しない。入口では詳細カタログを複製せず、深い SSOT へ誘導する
- **why を現在形で書く**: コードコメントを含め、現在の成立理由を書く。Issue / PR 番号や変更経緯は git / PR に任せる（外部 upstream 参照は `repo#NNN` 形式可）

### 手順（下書き → 基準）

1. 上の基準を読む。文書の都合で基準を緩めない
2. 節単位で書き直して基準へ適合させる。太字剥がし・言い換え削除だけの diff は不合格
3. 大型 reference（おおよそ 300 行超）は第 1 パスのあと、見出し一覧と壊れた表 / fence / 目次を再読する第 2 パスを必須にする
4. 完了条件で自己判定する。非自明な境界の API から動く正例が消えていないかも見る
5. プロジェクトに Markdown lint（目次・識別子 backtick 等）があれば通す

**品質パス完了条件**: 必要な判断情報の被覆を維持または改善し、より短く明確になっていること。行数だけが減った変更や構造が壊れた変更は未完了。

## 仕上げ: project docs 更新判定

確立済みのプロジェクト固有知見の永続化判定と、触った docs 周辺の削除・最新化。新しい設計原則の発見器ではない。追加の前に削れる箇所を先に洗う。観点は `references/review-prompt.md`。

**対象外**: `~/.agents/` 配下への project 知見の graduate（global への昇格は AGENTS の graduate 規則。dotfiles は提案のみ）。

| 種類 | 何を書くか |
| --- | --- |
| AGENTS.md | 概要・コマンド・スキル一覧 |
| Rules | 常時適用の規約 |
| Skills | 特定タスクの手順・判断・リファレンス |
| ユーザー向け docs | README / docs サイト等。機能と矛盾していれば直す |

配置はプロジェクト規約に従う。

| 軸 | 意味 |
| --- | --- |
| 起点 | 今回触ったファイル |
| 調査 | 既存 AGENTS / rules / skills / ユーザー向け docs を照合。未変更も読んでよい |
| 更新 | 対応記述に加え、重複・陳腐化・不整合も直す |

起点外でも削れる・直せる記述は直す。テーマが完全に別の大規模再構成だけ別提案。起点と diff の食い違いは報告。混在 working tree は起点を絞る。

### 手順

1. `git status --short` で起点を確定する
2. 材料を整理する: **知見**は検証可能な罠・正解パターン。**設計上の選択**は「A ではなく B」の主張で照合対象。単独で一般原則にしない
3. `references/review-prompt.md` に起点・知見・選択を埋めて渡す。使えなければ自己レビュー
4. 妥当な更新は起点外でも適用する。agent-facing を更新したら同じコミット前に品質パスも通す
5. 更新・見送りを簡潔に報告する

**project 判定の完了条件**: 起点に対する更新要否を判定し、要なら適用済み、不要なら見送り理由が言えること。
