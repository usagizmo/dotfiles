ドキュメント更新が必要か評価する。変更はせず、評価と提案だけ返す。
diff・既存ドキュメント・下記材料のみで判断する。

## 材料

| 材料 | 扱い |
| --- | --- |
| 起点一覧・diff・既存ドキュメント | 検証可能な事実。矛盾時の優先根拠 |
| セッション中の知見 | 記録候補のヒント。単独で一般原則へ昇格しない |
| 設計上の選択 | 親の主張。コード・既存規約・反復可能性と照合 |

## 起点
{親が確定したファイルパス一覧}
一覧と diff が食い違う場合は一覧を正とし、食い違いを報告する。

## 方針
- 既存 AGENTS / rules / skills / ユーザー向け docs は未変更でも照合してよい
- 起点外の重複・陳腐化・不整合・より良い書き方も更新案に含める
- テーマが完全に別の大規模再構成だけ別作業向け提案
- `~/.agents/` は対象外

## セッション中の知見
{検証可能な罠・ハマり・正解パターン。なければ「特になし」}

## 設計上の選択
{「A ではなく B。理由は〜」。なければ「特になし」}

## 変更の把握
1. `git status --short`
2. 起点の tracked は `git diff --cached` と `git diff`
3. 起点の untracked は中身を読む
4. diff が空で親が「直近コミットが対象」と明示したときだけ `git diff HEAD~1`

## 確認

### 0. 削除候補（追加より先）
- 重複 → SSOT 1 箇所 + 最小参照
- 経緯説明 → 現状規約だけ残す
- 上位原則に集約できるチェックリスト・表
- 解消済み制約・廃止 API
- 実害なし / 上位ルールでカバー済みの caveat

### 1–4. 更新要否
AGENTS.md / Rules / Skills / ユーザー向け docs。機能と矛盾していれば直す提案。

### 5. 確立済み知見の永続化
新しい原則の発見器ではない。既に確立したプロジェクト固有知見を残すかの判定。

残してよい: 反復確認されたパターン、検証可能な指針、既存 rule/skill では表せない再利用可能なドメイン知識、判定フロー・罠。

残さない: 一回限りの判断、未検証の一般化、コードや既存 docs から自明なこと、一般知識、プロジェクト横断の汎用原則。

記録先: 既存 rule / skill / references へ追記を優先。新規 references は置き場がなく複数タスクで再利用できるときだけ。書き方は `docs` skill の 品質パス（基準 + 手順）に従う。

## 更新原則
1. 削除を先に
2. 実装と二重管理にならないよう要点のみ
3. 古い情報・移行 note・経緯は削除
4. より良い構成があれば再構成を提案

## 出力
- 削除候補: パス・行・理由。なければ「なし」
- 更新が必要なファイル: パス・変更内容。なければ「更新不要」
- 不確実な追加は「提案（要確認）」
- 大規模再構成だけ「別作業向け提案」
