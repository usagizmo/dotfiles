ドキュメント更新が必要か評価してください。この段階ではドキュメントを変更せず、評価と提案だけを返してください。
実装者の暗黙知を前提にせず、diff・既存ドキュメント・以下に明示された材料のみで判断してください。

## 材料の扱い

| 材料 | 扱い |
| --- | --- |
| 起点一覧・diff・既存ドキュメント | 検証可能な事実。矛盾時の優先根拠 |
| セッション中の知見 | 記録候補を探すヒント。これ単独で一般原則へ昇格しない |
| 設計上の選択 | 親の主張。コード・既存規約・反復可能性と照合する |

## 起点（authoritative）
{親が確定したファイルパス一覧}
レビューの開始点。一覧と diff が食い違う場合は一覧を正とし、食い違いを報告してください。

## 調査と更新の方針
- 既存の AGENTS / rules / skills / ユーザー向けドキュメントは、未変更でも読んで照合してよい
- 起点外でも、重複・陳腐化・不整合・より良い書き方を見つけたら更新案に含める（無視しない）
- テーマが完全に別で大規模な再構成だけ「別作業向け提案」に分ける
- ホーム側 `~/.agents/` は対象外

## セッション中の知見（ヒント）
{検証可能な罠・ハマり・正解パターン。なければ「特になし」}

## 設計上の選択（主張・照合対象）
{「A ではなく B を選んだ。理由は〜」。なければ「特になし」}

## 変更内容の把握方法
1. `git status --short` を実行する
2. 起点内の tracked 変更は `git diff --cached` と `git diff` で把握する（パスを絞れるなら絞る）
3. 起点内の untracked ファイルは中身を直接読む
4. diff が空で、かつ親が「直近コミットが対象」と明示している場合のみ `git diff HEAD~1` を使う

## 確認対象

### 0. 削除候補（追加より先）
今回触ったドキュメント周辺を読み直し、追加の前に削減で済まないかを検討する:

- **重複**: 同じ概念が複数ファイル → SSOT 1 箇所 + 最小限の参照
- **経緯説明**: 現状規約だけ残せば足りる歴史的経緯
- **冗長な列挙**: 上位原則に集約できるチェックリスト・表
- **古い前提**: 解消済み制約・廃止 API
- **保険的 caveat**: 実害なし / 上位ルールでカバー済み

### 1. AGENTS.md
コーディング原則・新パターン・開発コマンド・スキル一覧など、プロジェクト全体に関わる変更。

### 2. Rules
新規・変更・削除が必要な規約。

### 3. Skills
description の適用タイミング、手順・形式、リファレンス。新規 skill が必要か。

### 4. ユーザー向けドキュメント
README / docs サイト / changelog 等。機能変更と矛盾していれば直す提案をする。プロジェクト固有のドキュメント規約があればそれに従う。

### 5. 確立済み知見の永続化判定

新しい原則の発見器ではない。**既に確立した**プロジェクト固有知見を、既存ドキュメントへ残すかの判定器である。

**残してよい**
- セッションで反復確認された、または明示的に確立したプロジェクト固有パターン
- 次回同じ間違いをしないための、検証可能な指針
- 既存 rule / skill では表せず、将来の複数タスクで再利用できるドメイン知識
- 判定フロー（いつ・何を使うか）、確立した罠・ハマり

**残さない**
- 一回限りの判断・セッション固有の好み
- 親の要約にしかない未検証の一般化
- コードや既存 docs から自明なこと、既に書いてある原則の繰り返し
- 一般知識としての標準パターン
- プロジェクト横断の汎用原則（`~/.agents/` 向け）

**記録先**
- 既存の rule / skill / `references/` へ追記・更新を優先（新規ファイルを先に作らない）
- 新規 `references/` は、既存の置き場がなく将来の複数タスクで再利用できるときだけ提案
- 書き方は「〜する方が美しい。なぜなら〜」を優先。具体例は既存スタイルに合わせる

## 更新原則
1. **削除を先に**: 削除候補を出し切ってから追加する。`+` ばかりなら設計を疑う
2. **簡潔に**: 実装と二重管理にならないよう要点のみ
3. **最新化**: 古くなった情報・移行 note・歴史的経緯は削除
4. **大胆に**: より良い構成が思いついたら再構成を提案する（関連する周辺 docs も含む）

## 出力形式
必ず次を含める:
- **削除候補**: パス・行・理由。なければ「なし」
- **更新が必要なファイル**: パス・変更内容（起点外の改善も含む）。なければ「更新不要」
- 不確実な追加・新規ファイルは「提案（要確認）」
- テーマが別の大規模再構成だけ「別作業向け提案」
