## 変更内容の把握方法
`git diff --cached` と `git diff` を実行。
diff が空なら `git log -1 --format="" --name-only` で直近コミットの変更ファイルを確認し、`git diff HEAD~1` で差分を取得。

## 確認対象

### 0. 削除候補（追加検討より先に必ず行う）
ドキュメントは追記バイアスで密度が下がりやすい。今回触ったドキュメント周辺を読み直し、以下を先に列挙する。
追加が必要に見える場合も、まず削減で済まないかを検討する:

- **重複**: 同じ概念・規約が複数ファイルに現れていないか（実装か該当 skill 1 箇所を SSOT にし、他は最小限の参照に置き換える）
- **経緯説明**: 「以前は〜」「移行のため〜」など、現状規約だけ残せば不要な歴史的経緯
- **冗長な列挙**: チェックリスト・テーブル列挙が増えすぎて上位原則に集約できる箇所
- **古い前提**: 既に解消された制約・廃止された機能・変更前の API への言及
- **念のための caveat**: 実害がない・既に上位ルールでカバー済みの保険的記述

### 1. プロジェクト AGENTS.md
プロジェクト全体に関わる重要な変更:
- コーディング原則・規約の変更
- 新しいパターンやルールの追加
- 開発コマンドの変更
- スキル一覧の更新

### 2. プロジェクト Rules（`.agents/rules/*.md`）
- 新しい規約の追加が必要か
- 既存の規約に変更が必要か
- 古くなった規約の削除

### 3. プロジェクト Skills（`.agents/skills/`）
- description の適用タイミングが正確か
- 手順や形式に変更が必要か
- 新しいスキルの追加が必要か
- リファレンスの更新が必要か

### 4. セッション知見の記録先判断

プロジェクト固有の規約・パターンだけをプロジェクト rules / skills に記録する。該当しないものは記録不要。

### 5. 設計振り返り（Design Retrospective）

セッション全体の変更を俯瞰し、設計レベルの学びを抽出してください。
（`/code-review` のような個別コードレビューではなく、セッションから蒸留すべき汎用的な設計原則の発見）

#### 問い
以下の観点について、今回のセッションで「美しい設計」に近づいた判断、または離れた判断があったか振り返ってください:

- **責務分割**: モジュール・関数の責務は単一か。「〜と〜をする」になっていないか
- **データフロー**: データの変換経路は最短か。状態の持ち主は適切か（SSOT）
- **抽象化レベル**: 粒度は適切か。同じレイヤーに同じ抽象レベルの処理があるか
- **パターン整合性**: 既存パターンと整合しているか
- **API 設計**: 呼び出し側にとって自然なインターフェースか

#### 蒸留すべきもの
- 「なぜこの設計が美しいか」の原則化（回避的ではなく追求的に記述）
- プロジェクト固有の設計美学
- アンチパターンとその理由（なぜダメか、何が代わりに正解か）

#### 記録先
プロジェクト固有の設計知見（パターン・アンチパターン・ドメイン知識）は、該当する `.agents/skills/*/references/` に記録する（既存ファイルがあれば追記、なければ新規作成）。

#### 書き方
- 「〜してはいけない」ではなく「〜する方が美しい。なぜなら〜」の形式を優先
- 具体例は ✅ Good / ❌ Bad の対比（既存スタイルに合わせる）

## 更新判断基準

**記載する:**
- プロジェクト固有の設計判断とその理由
- 判定フロー（いつ・何を使うか）
- 試行錯誤の末に確立したパターン（罠・ハマりポイント含む）
- エージェントが次回同じ間違いをしないための指針

**記載しない:**
- エージェントが一般知識として持つ標準パターン
- コード・型定義から自明に読み取れる情報や一般的な技術説明
- セッション固有の一時的な判断（パターンとして確立していないもの）
- 既に記録済みの原則の繰り返し
- プロジェクト横断の汎用原則（グローバル `~/.agents/` 配下はこの評価の対象外）

## 更新原則
1. **削除を先に**: 確認対象 0 の削除候補を出し切ってから追加を検討する。差分が `+` ばかりになっていたら設計を疑う
2. **簡潔に**: 実装と二重管理にならないよう、要点のみ
3. **最新化**: 古くなった情報・項目はすべて削除（移行 note・歴史的経緯も含む）
4. **大胆に**: より良い構成が思いついたら積極的に再構成する

## 出力形式
以下の 2 セクションを必ず含める:
- **削除候補**: ファイルパス・行・削除理由を具体的に。なければ「なし」と明記
- **更新が必要なファイル**: パス・変更内容を具体的に。なければ「更新不要」と明記
