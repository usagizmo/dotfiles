# Codex と付け合わせる設計判断

設計判断（実装前のプラン、実装後の設計再検討の両方）について、**まず実行中のエージェント自身が判断を下し**、Codex のセカンドオピニオンと突き合わせてから進める。

## なぜこのスキルが必要か

一人で下した設計判断には、本人に見えない盲点が残る。前提の取り違え、考慮漏れのエッジケース、より筋の良い代替設計、見落としたトレードオフ — これらは「別の頭」に通すと表面化する。別プロセスの Codex に read-only で確認させることで、現在の会話コンテキストから距離を置いたレビューを取れる。サブプロセス側が同じ skill / rules を同じ深さで読んでいるとは限らないため、**評価の判断軸はプロンプトに明示的に書き込む**（下記参照）。

ただし polarity を取り違えない。**判断主体は常に実行中のエージェント**であり、Codex は検証者。目的は「Codex に従う」ことではなく、**自分の判断を Codex と突き合わせて 1 段強くする**こと。指摘は精査の対象であって、命令ではない。

評価の北極星は `tidy`（根本解決・クリーン化）と「ゼロベース設計」— 実装の経緯を一切忘れ、今日この要件を受け取ったトップエンジニアがゼロから設計したら辿り着く姿と一致しているか。

## いつ発動するか

**仕様・設計を深く検討する場合**に単発で使う。毎回のパイプラインではなく、軽微な変更・自明な実装では使わない。

- 新機能・新モジュールの設計方針を立てたとき（事前）
- リファクタリング・改善のアプローチ、バグ修正の原因仮説と修正方針で迷いがあるとき（事前）
- 複数の実装方法でトレードオフを比較するとき（事前）
- 機能実装・リファクタを書き終え、「ゼロベースで設計したら同じになるか」を問い直したいとき（事後）
- **作業中に湧いた疑問の確認**。実装の途中でも、設計・品質・不具合に関する疑問はその場で Codex に確認してよい

### 聞き先の切り分け — ユーザーか Codex か

- **ユーザーに聞く**: 機能・方針の意思決定。プロダクトの方向性・要件の解釈・スコープなど、正解が好みや事業判断に依存するもの
- **Codex に聞く**: トップエンジニアがゼロベースで考えれば行き着く類のもの。実装方法の筋の良さ、品質、不具合・エッジケースの検証など、技術的に収束する問い

> 事後の設計再検討は、表面的なクリーンアップ（命名・不要コード・スタイル）を担う `tidy` とは別軸。こちらは設計の骨格そのもの（責務分割・データフロー・抽象化）を問い直す。

## 実行手順

### 1. まず実行中のエージェント自身が判断を下す

Codex に丸投げせず、まず自分で判断を作る。これがレビュー対象になる。

**事前（プランニング）の場合** — 実装方針のドラフトを作る。最低限:

- 何を作る/変えるのか（要件・ゴール）
- どのファイル・モジュールをどう変えるのか
- 主要な設計判断とその理由
- 想定するエッジケース・懸念点（自分で気づいている範囲で）

**事後（設計再検討）の場合** — 実装の経緯を一切忘れ、「今日この要件を受け取ったトップエンジニアがゼロから設計したらどうなるか」を考え、現在の設計がその理想と一致しているか以下の観点で自身が評価する（コードとコンテキストを深く読んでいる強みを活かす）:

- **責務分割**: 各モジュール・関数は単一の責務を持つか。「何をするものか」を一文で説明でき、「〜と〜」が含まれないか
- **API 設計**: シグネチャは呼び出し側にとって自然か。引数の数・戻り値の型は過不足ないか
- **データフロー**: 変換経路は最短か。状態の持ち主は適切か。同じデータを複数箇所で保持していないか（SSOT）
- **抽象化レベル**: 抽象の粒度は適切か（過剰でも不足でもないか）。同じ抽象レベルの処理が同じレイヤーにあるか
- **パターン整合**: 既存パターンと整合しているか。同じ問題を別の方法で解決していないか
- **エッジケース**: 境界値・null/undefined・空配列・並行処理・エラーハンドリングは考慮されているか

事後評価では、対象を git で特定する:

- 直前の変更がある場合: `git diff --cached` と `git diff`
- diff が空の場合: `git log -1 --format="" --name-only` で直近コミットの変更ファイル
- ユーザーが対象を指定している場合はその指示に従う

変更ファイルだけでなく、関連するモジュール・型定義・呼び出し元も読み、全体像を把握してから評価する。

いずれの場合も、判断が曖昧なほど Codex のレビューも曖昧になるので、筋を通してから渡す。改善案は破壊的変更を恐れない（部分的パッチより根本的な改善を優先）。

### 2. プロンプトをファイルに書き出す

プロンプトにはコードや記号が混じり shell のクォートで壊れやすいため、**一時ファイルに書いてから stdin で渡す**。`/tmp/codex-consult-prompt.md` に以下の構成で書く:

```markdown
あなたは設計判断のレビュアーです。以下の判断をレビューし、セカンドオピニオンを返してください。
コードの変更はしないでください。分析と提案だけを返してください。

## 背景・要件
{何を作る/変えるのか、または何を評価し直しているのか。ゴールと制約}

## 関連ファイル
{Codex が読むべきファイルパス。Codex はリポジトリを read-only で読めるので、
 全文を貼らずパスと「どこに注目すべきか」を示す}

## 私の判断
{実行中のエージェントが立てた実装方針、または事後の設計評価。設計判断とその理由を含める}

## レビューしてほしい観点
最重要の判断軸（トップエンジニアの目線）:
- **ゼロベース設計か**: この設計は、実装の経緯を忘れてゼロから設計したときに辿り着く最適解と一致するか。既存実装に引きずられたパッチ的な設計になっていないか
- **根本解決か**: 症状を抑えるだけの継ぎ足しでなく、原因に対する抜本的な設計か。根本解決のためなら改修スコープを広げる前提でよい
- **クリーンか**: 互換のための温存コード・deprecated alias・古い実装・dead code を残す前提になっていないか。新設計でクリーンに置き換える方針か

加えて、責務分割 / API 設計 / データフロー / 抽象化レベル / パターン整合の各観点で、現在の判断がゼロベース設計と一致するか評価してください。
- 見落とし・考慮漏れのエッジケースはないか
- より筋の良い代替設計はないか（あれば理由とともに）
- 隠れたトレードオフ・リスクはないか

末尾に、判断への総合評価（このまま進めてよい / 修正を推奨 / 再考を推奨）を一言で。
重箱の隅つつき (nitpick) は不要。実装の成否を左右する指摘に絞ってください。
```

### 3. Codex に渡す

read-only サンドボックスで実行し、最終回答だけを `-o` でファイルに受け取る（思考ログが混ざらない）:

```bash
codex exec -s read-only -o /tmp/codex-consult-out.md - < /tmp/codex-consult-prompt.md
```

- モデルと reasoning effort はコマンドで指定せず、codex の config に従う。セカンドオピニオンは質が命なので effort を下げる方向の上書きはしない
- reasoning が深いと応答に 1〜数分かかることがある。実行前にユーザーへ「Codex に確認します」と一言伝えてから走らせる
- エラー時（未インストール・認証切れ等）の対応は `references/codex-exec.md` を参照

実行後、`/tmp/codex-consult-out.md` を読む。

### 4. 一致点・相違点を整理して統合する

Codex の回答を鵜呑みにも黙殺もせず、**自身の判断と突き合わせ**、各論点を 3 つに仕分ける:

- **一致**: Codex も同意 → 判断の確度が上がった。そのまま採用
- **相違・追加指摘**: Codex が別案・懸念を出した → 中身を精査する。妥当なら判断に取り込む。妥当でない（誤検知・前提のズレ）なら、なぜ採用しないかを言語化する
- **判断が割れる**: どちらの設計にも理がある／プロダクトの方向性に依存する → ユーザーに判断を仰ぐ

最終判断は必ず実行中のエージェントが下す。読み込んだコンテキストに基づいて Codex の指摘を退けてもよい（その場合は理由を明示）。

### 5. 統合した判断を提示する

ユーザーには以下を簡潔に示す:

```markdown
## 設計判断（Codex 確認済み）

{統合した最終判断（実装方針、または事後の設計評価と改善案）}

### Codex のレビューを反映した点
- {取り込んだ指摘とその理由}

### Codex の指摘で採用しなかった点（あれば）
- {指摘 → 採用しない理由}

### 判断を仰ぎたい点（あれば）
- {割れた論点と選択肢}
```

判断が割れる重要な相違があれば、そこはユーザーの決定を待ってから着手する。なければ統合判断で実装に進む（事後の設計再検討で改善案が出た場合は、ユーザーが承認した項目について実装に進むか確認する）。
