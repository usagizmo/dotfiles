# 設計判断の突き合わせ

**判断主体は実行中のエージェント**。アドバイザーは検証者（命令ではない）。北極星: ゼロベース設計 / 根本解決 / 互換コード排除。

## いつ

- **軽量**: 修正方針・短い技術的疑問（「これでいい？」）。自分の案は 3〜6 行でよい。承認ゲートなし
- **深い**: 新機能・大リファクタの設計方針、複数案のトレードオフ、実装後の骨格の問い直し
- プロダクト方針 → ユーザー。技術的収束 → アドバイザー。自明なら不要

## 手順

### 1. 自分で判断を書く

- **軽量**: 問い / 案 / 触るパス / 懸念（短く）
- **深い・事前**: ゴール / 変えるファイル / 主要判断と理由 / 懸念
- **深い・事後**: ゼロベースで理想形を考え、責務分割・API・データフロー・SSOT・エッジで現状を評価（`git diff` 等で対象を特定し関連コードも読む）

### 2. プロンプトを一時ファイルへ

`PROMPT=$(mktemp "${TMPDIR:-/tmp}/consult-prompt.XXXXXX")`。軽量なら観点を絞って短くてよい:

```markdown
あなたは設計判断のレビュアーです。コードは変更せず分析と提案だけ返してください。

## 背景・要件
{...}

## 関連ファイル
{パス + 注目点。全文は貼らない}

## 私の判断
{...}

## 観点
- ゼロベース設計か / 根本解決か / クリーンか（互換温存なし）
- 見落としエッジケース・より良い代替・隠れたトレードオフ
（軽量時は「致命的な穴と短い代替」に絞ってよい）
末尾: このまま進めてよい / 修正推奨 / 再考推奨（一言）。nitpick 不要。
```

### 3. アドバイザーに渡す

同ディレクトリの `advisors.md`（起動表・失敗時ポリシー。harness 差分はここだけ）に従う。モデル / effort は上書きしない。失敗は隠さない。

### 4. 統合して提示

一致 / 取り込み / 不採用（理由）/ ユーザー判断が要る点。最終判断は実行中のエージェント。軽量は 5〜10 行でよい。複数アドバイザーなら出典タグ（`advisors.md` の出典表記）を付す。

### 5. 実装プラン確定時のみ承認ゲート

軽量・単発の設計再検討では不要。プラン確定時は統合プラン（仕様・変更対象・判断・スコープ外・検証）を提示し、GO まで実装しない。
Plan Mode がある harness はそれを使う（例: `enter_plan_mode` → プラン文書 → 承認 UI）。無ければ構造化 Markdown + 明示 GO。

### 6. 実装へ進んだ場合の完了

consult 後に実装した場合の仕上げフローは `~/.agents/AGENTS.md`（仕上げ SSOT）に従う。consult 単体の終了で実装完了としない。
